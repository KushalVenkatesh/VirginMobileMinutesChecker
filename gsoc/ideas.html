Introduction

The Minutes Checker app performs a simple function: it tells users information about their Virgin Mobile cell phone account. It indicates how many minutes they have for the month and when their bill is due. This app has been installed over 45,000 times, has a 5-star rating (with 1200+ ratings) and I receive feedback from customers on a daily basis.

The development team consists of myself and a handful of developers who have submitted patches. This app is ideal for motivated students who have limited Android experience. The code base is small enough that one can easily grasp it and make changes.

You'll get experience exploring a new code base with multiple contributors - and hopefully help to normalize our coding styles and practices! You will touch all aspects of the application, from back-end networking code to front-end presentation.

Ideas

Most of these ideas have come directly from the aggregate of customer feedback. Others are more experimental in nature.

1. User Experience and Design

The user experience leaves a lot to be desired. Specific complaints are:

  1. The app lets one track multiple accounts; how do I delete an account?
  2. Can we make the widget more intuitive? Can it look better? Convey more useful information?
  3. The app does not adjust to different-sized screens. Users with large screens still get information rendered in small fonts.
The app does have a "help" page, but we should have a UX which uses common affordances and does require the user to read a page of instructions.

Related to this is graphic design. One gentleman has contributed gorgeous mock-ups for this app. Let's make them a reality.

This project would be great for a student who has experience in UX. You would be responsible for designing a user flow, wireframing it, and then writing code to make it a reality. This probject would not require as much work on the back-end (eg, networking code.) Ideas for UX improvements are not limited to what has been described here; the broad goal is to improve on what we have now.

It would be great to push out changes (yes, to all users) and elicit feedback, and iterate a few times before the summer ends.

Mentor: Jessica Graham

2. Back-end architecture

As the app has evolved, we have not really spent much time on architecture. How should we organize the network-related code so that, when Virgin Mobile changes the information they provide, we can easily fetch those changes? How should we store user's information, or information for multiple accounts?

This also entails planning how we should propagate information to the front-end. How can we achieve a better separation between the "presentation" and and the back-end?

Your task would be to take this app's requirements, understand the kinds of changes that are likely to occur in the future, come up with a design, and then implement. We should have a standard way to cope with changes, as well as conventions for code style.

There are no specific ideas for how to do this; we want you to come up with a proposal and carry it through! Possibly work with one of us as you write your app to create a more concrete plan of attack.

Mentor: Ken Ganong, Jay Goel

3. User-requested features

There are several new features that users have been requesting sicne the beginning. We should implement them!

  1. Handle multiple types of accounts. Currently this app only displays information for VM's "Beyond Talk" plan, but some families use PayLo or Broadband2Go plans. Can we support this?
  2. Users also want to be able to see their call log and txt message usage.
  3. Starting in March, VM will cap data usage at 2.5GB. We should help users with this as well.
  4. Have an option for other kinds of alerts and messages when they are running out of minutes.

These are fairly straightforward tasks and would go a long way to make this app useful.

Mentor: Ken Ganong, Jay Goel

4. Testing

We do not have a great infrastructure for testing. It would be nice to create mock users or mock HTML so we don't have to hit the VM website or use our personal information when we want to test. We should have JUnit tests and then a plan for testing the GUI. There are bugs that we would have been able to solve sooner with testing. How can we test for when a user's network connection drops? How can we monitor the CPU and battery utilization of the background service?

In a similar vein, can we do a better job eliciting user feedback? Ask or survey them, within the app, about what features they want? Provide news updates to them? This is not strictly in the scope of "testing" but it is relevant to getting feedback about things which are broken so that we can improve going forward.

You would be responsible for coming up with different areas and scenarios to test, and then implementing a framework for doing this. The project would not encompass all of the above; you would come up with what you think the most effective attributes for testing are and implement it.

Mentor: Ken Ganong, Jay Goel

5. Ads

Finally, this app uses AdMob for ads. The ads are an experiment - how do users react? They also help offset some of the costs - the Android account, maintaining github, purchasing devices for testing, etc.

There are other companies which serve ads. Can we easily switch ad providers? What about selling ads directly, rather than going through an agency? We could do something similar to what StackOverflow does, and do free ad placement for open-source apps.

Are there different form factors or ad layouts which make for better advertising? Can we use other user preferences for show better ads?

Ads are an experiment. It would be fun and informative to play with these attributes and see what we can learn.

Mentor: Ken Ganong, Jay Goel
